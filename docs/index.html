01

先看全局

你最终会得到什么

Resume Skills 不是一个在线填表网站，而是一组让 AI 按稳定规则工作的 Agent Skills。它将内容事实、视觉母版、岗位定制和版本记录分开，避免每投一个岗位就重新做一份简历。

工作区为空

调用 Workflow 建立母版

上传旧简历或从零接受采访，确认事实，选择输出模式与模板，生成可长期维护的母版。

$resume-workflow

已经有母版

直接输入目标 JD

先看匹配证据与真实缺口，再确认变更预览，生成独立的公司/岗位版本。

母版 + JD → 定制版
输入旧简历或真实经历
确认resume-facts.yaml
生成简历母版
投递JD 岗位版本
事实边界

JD、模板示例和 AI 建议都不是你的经历。任何技能、数字、职责或成果只有在你确认后，才能进入最终简历。

02

准备环境

安装全部 Skills

建议一次安装完整技能集，因为 JD 定制、ATS 审计和版本管理会共享事实契约与写作规范。安装 Skill 不要求先安装 npm 包；npm CLI 只用于打开本地 Canvas。

2.1 检查 Node.js

需要 Node.js 20 或更高版本。打开 PowerShell、Windows Terminal 或其他终端：

PowerShell / Terminal
node --version
npm --version

如果第一条命令显示 v20.x 或更高版本，可以继续。若提示命令不存在，请先从 Node.js 官方网站安装 LTS 版本，然后重新打开终端。

2.2 安装全部 Skills

安装到当前项目
npx skills add Chasen-Liao/resume-skills
全局安装示例
npx skills add Chasen-Liao/resume-skills --skill '*' --global --yes

2.3 确认安装结果

安装后重新打开或新建 Agent 会话。你应该能够显式调用 $resume-workflow。如果客户端没有自动触发，直接在提示词里写出 Skill 名称即可。

03

保护隐私

建立私有简历工作区

真实简历包含手机号、邮箱、学校和经历，不要把它们放进 Resume Skills 的公开源码仓库。请在项目目录外创建一个独立文件夹，后续事实库、母版和 JD 定制版都保存在这里。

Windows PowerShell 示例
New-Item -ItemType Directory -Path "D:\MyResume" -Force
Set-Location "D:\MyResume"
不要公开个人信息

默认只在本机保存。即使以后使用 Git，也先使用本地私有仓库；除非你已经清理所有隐私字段，否则不要推送到公开 GitHub 仓库。

推荐目录结构

MyResume/
├── resume-facts.yaml            # 唯一事实源
├── master/
│   ├── resume-visual.html       # 视觉母版
│   ├── resume-visual.pdf
│   ├── resume-ats.html          # 按需生成
│   └── resume-ats.pdf
└── tailored/
    └── company-role/
        ├── jd.md
        ├── resume-visual.html
        ├── resume-ats.html
        ├── matching-analysis.md
        └── version-notes.md
04

第一次使用

在空工作区调用 Workflow

当目录中还没有 resume-facts.yaml 和母版时，从 $resume-workflow 开始。Workflow 会判断你是导入已有简历还是从零接受采访。

路线 A：你已经有旧简历

复制给 Agent
使用 $resume-workflow 帮我建立简历母版。
我会提供一份已有简历，请先解析并列出：
1. 已识别且需要我确认的事实；
2. 模糊、冲突或可能过期的信息；
3. 最近经历、项目和技能中需要补充的内容。

请一次只问一个最重要的问题。不要把解析内容直接当作事实，
也不要编造经历、技术、数字或成果。

可提供 PDF、DOCX、HTML 或可复制文本。扫描 PDF 或 OCR 结果可能不准确，Agent 应标记不确定内容，并要求你确认或提供可复制文本。

路线 B：你没有任何简历

复制给 Agent
使用 $resume-workflow 从零帮我建立简历母版。
请逐步采访我，从基础信息开始，一次只问一个问题。
优先追问真实角色、行动、交付物和可核验结果。
没有可信数字时不要替我估算，也不要为了填满模板编内容。

Workflow 应该如何提问

  1. 1
    先问最近变化

    例如新项目、实习、比赛、开源贡献、课程成绩或技能变化。

  2. 2
    再问影响最大的缺口

    例如项目角色不清、结果没有证据、时间冲突或技能熟练度缺少依据。

  3. 3
    最后补齐基础字段

    联系方式、教育时间、链接和求职方向等投递必需信息。

05

唯一事实源

确认 resume-facts.yaml

resume-facts.yaml 是整套流程最重要的文件。它不是视觉简历，而是一份可追溯的事实库：每条对外陈述都记录来源、证据、置信度和确认时间。

事实记录示例
facts_version: 1
claims:
  - claim_id: "project-rag-01"
    section: "projects"
    claim: "实现课程资料的检索增强问答原型"
    source: "user"
    confidence: "high"
    evidence:
      - "候选人确认独立完成检索与生成链路"
    metric_status: "not_applicable"
    confirmed_at: "2026-07-16"

pending_claims: []
字段意义进入最终简历的要求
source来自用户、旧简历或可核验材料来源明确
confidence当前证据是否充分且一致低置信度必须回访
evidence为什么可以这样写至少有可理解的依据
metric_status数字是否已核实未验证数字不能进入成稿
confirmed_at用户最后确认时间用于判断信息是否过期
pending_claims 的作用

旧简历解析出的模糊内容、JD 中缺失的要求和未经确认的改写建议可以暂存在这里，但绝不能复制进最终 HTML/PDF。

06

内容确认后再选样式

选择输出模式与模板

事实确认完成后，先选择输出模式。如果选择视觉版，再从六套 A4 模板中选择一种。视觉选择不应改变事实内容。

视觉 HTML/PDF

适合人工阅读

  • 更强的层级、色彩和版式
  • 可进入本地 Canvas 微调
  • 适合作品集、邮件和直接投递

ATS-safe HTML/PDF

优先机器解析

  • 单栏、标准标题、纯文本顺序
  • 减少复杂表格与装饰
  • 不能保证通过任何 ATS

六套视觉模板

现代简约留白、高对比、克制
经典商务深蓝、稳重、专业
创意个性大胆配色、强辨识度
日式极简暖色、轻质、呼吸感
科技感深色、高对比、终端美学
简约蓝色商务清晰、克制、通用
选择示例
事实内容已经确认。请生成视觉 HTML/PDF 母版，
使用“现代简约”模板，并同时保留 ATS-safe 版本。
两种输出必须使用同一份 resume-facts.yaml，不要改变事实。
07

最后一公里

预览、微调与导出 PDF

resume-builder、jd-tailorer 或 resume-workflow 生成视觉母版或视觉定制版并完成 PDF 验证后，交付流程必须启动本地 Canvas，让用户先看到最终文件并检查 A4 页面。Canvas 只负责已有文字和排版，不负责 AI 改写、JD 匹配或结构重排；若当前环境不能执行 npx，Agent 必须明确说明未启动并给出带实际 HTML 路径的完整命令。

在简历目录中运行
npx @chasen-liao/resume-skills editor resume-visual.html

可以调整

  • 已有文字
  • 字号、字重、颜色、对齐
  • 行高、段后距、页边距和主题色

不支持

  • 凭空添加或改写经历
  • 自由拖拽与结构重排
  • 图片编辑与 JD 匹配
  1. 1
    打开视觉 HTML

    命令会启动本地服务并尝试自动打开浏览器。

  2. 2
    检查溢出与可读性

    出现溢出时先删减低相关内容或调整间距，不要压缩到难以阅读。

  3. 3
    导出编辑后的 HTML

    固定生成 <原文件名>-edited.html，不覆盖原文件。

  4. 4
    浏览器打印 PDF

    纸张选择 A4，按需要关闭浏览器默认页眉和页脚。

08

已有母版后的日常入口

输入 JD，生成岗位简历

有了 resume-facts.yaml 和母版后，不需要再次调用完整采访流程。把目标 JD 交给 Agent，它会先分析要求，再展示变更预览，最后生成独立岗位目录。

岗位定制提示词
我已有 resume-facts.yaml 和简历母版。
请分析下面的 JD，先给我：
1. 必需条件、加分项和岗位职责；
2. 我的直接匹配、相关证据、真实缺口和待确认项；
3. 定制版的变更预览，包括前置、弱化和措辞调整。

等我确认后再生成定制版。不要把 JD 要求当成我的经历。

[在这里粘贴 JD]

你应该先看到变更预览

JD 关注点候选人事实计划调整
React 性能优化项目中有首屏优化证据前置项目并突出已确认结果
LLM 应用开发事实库中有 RAG 项目使用一致术语对齐 JD
推荐系统经验没有相关 claim保留为真实缺口，不写入简历
确认后才生成

你可以拒绝某项措辞或排序。定制版只能重排、强调、压缩或对齐已有事实，不能新增技能和成果。

09

投递前质量关卡

运行 ATS 审计

ATS 审计会把“可以修复的呈现问题”和“不能伪造的事实缺口”分开。它用于降低机器解析风险，不保证通过筛选，也不代表录用概率。

A

解析与格式

检查可提取文本、标准章节标题、正文顺序、复杂表格、图片文字和页数。

B

关键词证据

区分精确匹配、相关表达与缺失项，并追溯到事实库和 JD 来源。

C

修复建议

呈现问题可在确认后修复；真实能力缺口只能追问或保留。

ATS 审计提示词
请对这份岗位定制简历做 ATS 审计。
将问题分成：
1. 可以直接修复的格式与解析风险；
2. 需要我确认的事实；
3. 必须保留的真实能力缺口。

不要堆砌关键词，也不要承诺一定通过 ATS。
10

可追溯、可回滚

用 Git 管理母版和投递版本

Git 很适合记录事实库和岗位版本的变化，但所有初始化、提交和推送都应由你明确确认。通常一个本地私有仓库就足够；每个 JD 用独立目录和提交记录，不必为每家公司创建分支。

首次初始化（确认后执行）
git init
git add resume-facts.yaml master
git commit -m "建立简历事实库与母版"
生成岗位版本后
git add tailored/company-role
git commit -m "定制：公司名 - 岗位名"

什么时候才需要分支

只有当你要同时试验互斥的母版方向，例如“前端工程”和“AI 产品”两套长期定位时，才考虑分支。普通岗位定制留在 tailored/ 并提交即可。

远程仓库必须谨慎

如果确实需要跨设备同步，请使用私有仓库，并再次检查提交中是否包含手机号、邮箱、住址、证件或其他敏感信息。

11

长期维护

更新母版并继续投递

以后有新实习、新项目或技能变化时，先更新事实库与母版，再生成新的 JD 版本。不要直接在旧岗位版本上累积事实，否则多个版本会逐渐漂移。

更新母版提示词
使用 $resume-workflow 更新我的简历母版。
请先读取现有 resume-facts.yaml，询问我最近发生的变化，
只对新增、变化或可能过期的字段做增量追问。
等我确认后再更新事实库和母版，不要覆盖历史岗位版本。

推荐维护节奏

  • 经历发生变化时立即补充事实和证据，避免几个月后忘记细节。
  • 开始一轮求职时确认联系方式、链接、求职方向和最近经历。
  • 每次正式投递前检查 JD 变更预览、ATS 报告、PDF 页数和文件名。
  • 收到面试反馈后可以调整表达与排序，但不能反向虚构事实。
12

遇到问题先看这里

常见问题与排查

Agent 没有自动调用 resume-workflow

重新打开会话，并在提示词中显式写出 $resume-workflow。还要确认安装时使用了 --skill '*'，而不是只安装单个 skill。

旧 PDF 解析出来的时间或数字不准确

不要直接进入最终简历。把它放在待确认列表，提供可复制文本或由你逐项确认。扫描件和 OCR 内容默认需要更严格复核。

AI 一直追问数字，但我没有数据

明确告诉它使用真实的角色、范围、交付物、上线/验收状态等非数字证据。没有可信数字时不应估算。

生成的 PDF 超过一页

先删减低相关内容，再压缩间距和行高。不要删除关键事实，也不要把正文缩小到难以阅读。资深候选人可根据实际情况保留合理页数。

Canvas 无法打开 HTML

Canvas 只支持带 Resume Skills 编辑协议的内置视觉模板。ATS-safe HTML 或外部模板可能无法打开，可直接用浏览器查看并打印。

JD 要求我没有的技能，应该怎么办

如实标记缺口，不要写入技能栏。可以展示相关证据，但要明确它只是相关而非直接匹配。

是否应该把整个简历仓库推到 GitHub

默认不建议。GitHub Pages 站点可以公开，但真实简历工作区应保持本地或使用私有仓库，并排除敏感信息。
