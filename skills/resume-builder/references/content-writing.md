# 简历内容写作要点

基于 2025-2026 年招聘趋势和 ATS 筛选现状整理的核心写作规范。

---

## 关键数据

| 数据 | 含义 |
|------|------|
| 87% 企业使用 ATS 筛选 | 简历先过机器再过人 |
| 仅 35% 简历通过初筛 | 60%+ 候选人在简历环节被淘汰 |
| HR 平均阅读 6-15 秒 | 「黄金 3 秒法则」— 前 1/3 页面决定命运 |
| 2025 届毕业生 ~1158 万人 | 竞争空前，差异化至关重要 |

---

## 一、STAR 法则写作公式

每一条经历/项目描述必须遵循 STAR 结构：

```
强动词 + 岗位关键词 + 量化结果
```

| 要素 | 含义 | 示例 |
|------|------|------|
| Situation | 背景/问题 | "负责公司内部管理系统" |
| Task | 目标任务 | "需要提升数据查询性能" |
| Action | 具体动作 + 关键词 | "使用 Redis 缓存 + SQL 优化重构查询层" |
| Result | 量化成果 | "查询响应时间从 3s 降至 200ms，QPS 提升 10 倍" |

### 对比示例

| 弱（职责描述） | 强（STAR + 量化） |
|---------------|-------------------|
| "负责社团公众号运营，发布多篇文章" | "独立运营学院公众号（粉丝 3000+），通过选题优化 + 封面 A/B 测试，3 个月内平均阅读量提升 120%，单篇最高转化 27 人报名活动" |
| "负责管理销售团队" | "带领 10 人销售团队连续两季度超额完成目标 15%，团队人效排名区域第一" |
| "参与了商城项目开发" | "主导商城订单模块开发，使用 React+Node.js 搭建，上线后日均处理订单 2000+，支付成功率 98.5%" |

## 二、数据量化对照表

将模糊描述替换为可量化数据。**每一条经历至少包含一个数字。**

| 模糊描述 | 量化替代 | 追问方式 |
|----------|----------|----------|
| "很多" | "200+" / "5000+阅读" | 具体是多少？ |
| "显著提升" | "提升 40%" / "增长 1.5 倍" | 提升了多少？ |
| "高效完成" | "3 天内完成" / "比预期提前 1 周" | 用了多长时间？ |
| "获得好评" | "满意度 95%" / "获学院推荐" | 如何证明？ |
| "参与项目" | "主导/负责 XX 模块" / "独立完成" | 你的角色是什么？ |
| "使用多种技术" | "React / Node.js / PostgreSQL" | 具体是什么？ |
| "团队合作" | "与 5 人团队协作" / "跨部门协调 3 个团队" | 多少人？什么角色？ |
| "提高了效率" | "自动化 80% 手工流程" / "节省 15h/周" | 提高/节省了多少？ |

收集信息时主动追问："这个成果有没有具体数字？提升了多少？涉及多少人？用了多长时间？"

## 三、ATS（应聘追踪系统）兼容性 ⚠️ 关键

87% 的企业使用 ATS 自动筛选简历。如果简历无法被 ATS 正确解析，HR 根本看不到。

### ATS 绝对避免

| ❌ 避免 | ✅ 替代 |
|--------|--------|
| 多栏/双栏布局 | **单栏从上到下**结构 |
| 图片、图标、Logo | 纯文字 + 标准项目符号 |
| 表格、文本框 | 简单章节标题分隔 |
| 页眉/页脚放关键信息 | 所有内容放在正文 body 中 |
| 特殊符号、emoji | 标准文字 |
| 花哨字体 | 系统字体（微软雅黑/Arial） |
| PDF（部分旧 ATS） | **.docx 最安全**（HTML 导出后建议同时提供 Word 版本提示） |

### ATS 友好标题（必须使用标准命名）

```
个人信息 / 求职意向
教育背景
实习经历 / 工作经历
项目经验
技能
荣誉奖项 / 竞赛获奖
自我评价
```

不要用创意标题（如"我的故事""为什么选我"），ATS 无法识别。

### ATS 自检方法

1. 从生成的 HTML 简历中复制全部文本
2. 粘贴到纯文本编辑器（记事本/Notepad）
3. 检查：
   - 所有信息是否完整出现？
   - 阅读顺序是否正确（从上到下）？
   - 有没有乱码或错位？
4. 如果有问题 → 说明 ATS 解析会出问题

### HTML 简历的 ATS 折中策略

HTML 简历主要用于**视觉效果 + 打印 PDF**。为覆盖 ATS 投递场景：

- 布局优先选**单栏 + 双栏混合**：顶部信息区单栏，主体内容单栏，仅在侧边栏（技能/联系）区域使用窄双栏，且确保这些信息在主内容区也有体现
- 技能标签使用语义化 `<ul>` 和 `<li>` 列表配合 CSS 伪元素分隔（`·`）而非独立色块，确保 ATS 可正确切分词汇
- 章节标题使用标准 `<h2>` 标签 + 标准命名
- 最终建议用户：本 HTML 适合打印 PDF 和网页展示；如需投递 ATS 系统，可提供 `.docx` 版本

## 四、简历板块优先级排序（应届生版）

1. **个人信息** — 姓名、电话、邮箱、求职意向、GitHub/LinkedIn（不放年龄/婚姻状态）
2. **教育背景** — 学校、专业、学历、时间、GPA（≥3.5 才写）、核心课程（选 3-5 门相关）
3. **项目经验** — ⚠️ 建议前置到实习经历之前（应届生最强能力证明）
4. **实习经历** — STAR 法则 + 量化数据，每个经历 2-3 条
5. **技能** — 分"精通/熟练/了解"三级，只写与岗位相关的
6. **竞赛/荣誉** — 标注级别（国家级/省级/校级）和时间
7. **自我评价** — 50 字以内，三句话结构（见下）

---

## 五、自我评价三句话结构

控制在 50 字以内，用以下结构：

1. **专业定位**："XX 专业应届毕业生，主攻 XX 方向"
2. **核心能力**："具备 XX + XX + XX 能力"（用 JD 关键词）
3. **职业目标**："期望在 XX 领域深耕，为团队贡献 XX 价值"

> ❌ 空洞版："性格开朗，吃苦耐劳，具有团队合作精神，学习能力强..."  
> ✅ 简洁版："计算机科学专业应届生，主攻前端开发方向。具备 React 生态 + TypeScript + 工程化能力。期望在前端领域深耕，为产品团队带来高质量的用户体验。"

## 六、文件命名规范

```
姓名_岗位_学校_毕业时间.pdf
```

示例：`张三_前端开发工程师_北京大学_2026.pdf`

## 七、常见致命错误

| 错误 | 为何致命 |
|------|----------|
| 一份简历海投所有岗位 | 关键词不匹配，ATS 直接过滤 |
| 写「负责 XX 工作」无成果 | HR 看不到你的价值——改写成「通过 XX 动作，实现 YY 结果」 |
| 自我评价写空话套话 | 毫无区分度，HR 直接跳过 |
| 技能标签列 20+ 个不分类 | 看不到重点，建议分「精通/熟练/了解」三级，控制在 8-12 个 |
| 项目经历只写技术栈不写成果 | 「用 React 做了一个商城」→ 缺用户量/性能数据/业务成果 |
| 排版稀疏导致 2 页 | 应届生必须 1 页——精简内容或用紧凑排版修复 |
| 造假的经历或技能 | 面试追问细节会当场暴露 |

- [ ] 自我评价 ≤ 50 字？（三句话结构）

- [ ] 文件命名规范？
- [ ] 纯文本粘贴测试通过？（ATS 自检）
- [ ] 无表格/图片/特殊符号？
- [ ] 与目标岗位 JD 关键词有交集？
