# 简历内容写作要点

本文件与 [`resume-contract.md`](resume-contract.md) 配套使用，供 `resume-builder` 和 `jd-tailorer` 共同遵循。写作目标是让已知事实更清楚、更容易核验、更贴近目标岗位；不是把不完整的信息包装成完整经历。

## 一、证据优先，而不是数字优先

每条经历或项目描述都应先回答：候选人做了什么、在什么范围内做、交付或改变了什么、证据是什么。数字有帮助，但不是硬性门槛；不得为了满足格式补数字。

收集信息时优先追问：

- 是否有可核验的指标、起止范围、时间节点、性能或质量记录？
- 影响了什么对象、模块、流程或交付物？
- 候选人是独立完成、主导、负责模块，还是在团队中协作？
- 是否有上线、验收、测试、发布、获奖、复盘或其他可核验结果？

如果没有可信数字，可以使用已经确认的证据：

| 证据类型 | 可接受的写法方向 | 边界 |
|---|---|---|
| 规模/范围 | 部门级流程、校园项目、面向内部用户的系统、负责订单模块 | 不把模糊的“大量”改成具体数量 |
| 时间 | 在实习期间完成、按期交付、某学期上线 | 不推算起止日期或提前周期 |
| 质量/状态 | 通过测试、完成验收、上线运行、形成文档或组件 | 不把“参与”改成“显著提升” |
| 角色 | 独立实现、主导方案、负责模块、与团队协作 | 不把团队成果全部归为个人成果 |
| 交付物 | API、页面、数据管道、实验报告、运营方案 | 交付物必须确实存在且由来源支持 |

指标状态应按事实契约记录：已核实才写具体数字；候选人明确认可的估算须保留“约/估算”等限定；未验证数字先追问或删除。具体规则见 [`resume-contract.md`](resume-contract.md)。

## 二、句子结构和 STAR 的使用方式

推荐使用以下结构，按信息完整度灵活取舍：

```text
强动词 + 具体动作/技术 + 结果或证据 + 背景/范围
```

STAR 可以帮助发现背景、任务、行动和结果，但不是每条 bullet 的固定格式。没有结果数字时，写清楚真实行动、角色、交付物和已确认状态即可；不要把职责改写成未经证明的成果。

| 信息不足时 | 有证据时 |
|---|---|
| “参与了商城项目开发” | “负责商城订单模块，实现下单与支付流程并完成联调交付” |
| “负责公众号运营” | “独立完成选题、排版和发布，按月复盘内容表现并沉淀运营流程” |
| “使用多种技术” | “使用 React、Node.js 和 PostgreSQL 完成前后端功能开发” |

以上示例只展示表达结构，不代表任何候选人的事实。若有数字，应替换为有来源的实际指标；若没有，就保持非数字证据，不自行添加数量、比例或时间。

## 三、各板块写作规则

- `profile`：只放投递需要且用户确认的联系方式、所在地、求职意向和链接。链接应可访问；待确认链接只能留在采集/分析报告，不能进入最终简历成稿。
- `education`：学校、专业、学历和时间优先；成绩、课程和研究方向仅在有来源且与岗位相关时加入。
- `experience` / `projects`：突出真实角色、行动、技术、交付物和结果；将团队成果与个人贡献分开。项目经验是否前置由候选人阶段和岗位相关性决定，应届生通常优先展示项目，但不是事实规则。
- `campus`：记录校园组织、学生工作、志愿服务、社团、校园活动等经历的组织/活动、角色、时间、行动、交付物和结果；只写能由候选人或材料核验的个人贡献，不把集体活动自动写成个人成果。
- `skills`：只列实际使用或明确掌握的技能。熟练度必须有候选人依据，不用“精通/专家”等强标签替代证据；技能数量服从岗位相关性和版面，不设固定数量。
- `awards`：写清奖项名称、授予方、级别和时间（若已确认）；区分获奖、入围、参赛和获得证书。
- `summary`：从其他板块归纳职业定位、已证能力和求职方向；不设固定字数或固定句数，不写无法举证的性格、潜力和宏大承诺。

## 四、关键词与 JD 定制

每个关键词都应标注来源：

- `candidate`：来自候选人已确认的简历、经历或技能；可以作为事实使用。
- `jd`：来自目标职位描述；只能用于排序、术语对齐和匹配分析，不能证明候选人具备该能力。
- `user_preference`：用户指定希望采用的表达或语言；仍须通过候选人事实校验。

优先使用候选人已有事实中与 JD 一致的术语，在自然语境中放入经历和技能板块。不能为了提高匹配度堆砌关键词、重复同一术语，或把 JD 的职责直接复制成候选人成果。缺失要求应如实列为缺口，相关但较弱的证据可以降级呈现。

## 五、ATS 与两个输出模式

ATS 解析行为取决于具体系统、文件格式和模板，任何排版策略都不能保证通过筛选。应把“更容易解析”与“必然通过”区分开。

### 视觉 HTML/PDF 模式

- 目标是网页展示和浏览器打印 PDF；可使用品牌色、适度双栏、技能标签和视觉层级。
- 仍应使用真实文本、清晰的标题层级和合理的阅读顺序，避免把关键信息只放进背景图、图标或装饰元素。
- 生成后检查打印页数、裁切、复制文本、链接和中文字体；视觉效果不等于 ATS 兼容保证。

### ATS-safe 模式

- 目标是提高机器解析稳定性；优先单栏、标准章节标题、普通文本和清晰的时间/组织/职位关系。
- 避免用图片承载文字、复杂嵌套表格、文本框、页眉页脚中的关键信息、emoji/图标等装饰符号和难以复制的装饰字体；普通项目符号、连字符和技术名称可保留，但必须能被复制并保持文本顺序。
- 技能用可复制的文本列表表达；联系方式和重要经历放在正文阅读流中。
- 根据投递平台要求选择 HTML 或浏览器打印 PDF；仓库当前没有 DOCX 生成能力，不能承诺或输出 DOCX。若平台只接受 DOCX，应明确说明当前能力边界并由用户另行转换。

两个模式可以共享同一份事实数据，但不能为了视觉模式或 ATS 模式改变 claim 的内容和证据状态。

## 六、隐私字段与头像默认策略

### 中国中文简历

- 默认收集并展示：姓名、手机、邮箱、所在地/期望城市、求职方向，以及用户确认的作品链接。
- 默认不展示身份证号、家庭住址、婚姻状况、健康信息、银行信息等敏感字段；年龄、出生日期、民族、政治面貌等只有在用户明确要求且确认投递场景需要时才考虑。
- 头像默认不主动添加。只有在用户提供头像并明确希望展示，或用户确认目标场景确实需要时才放入；不能从示例资源、网络或其他文件推断使用哪张照片。

### 国际英文简历

- 默认收集并展示：姓名、专业联系方式、所在地（按投递需要）、作品/LinkedIn/GitHub 链接，以及用户确认的工作授权或签证信息。
- 默认不展示照片、年龄、出生日期、性别、婚姻状况、国籍、种族、身份证件信息和完整住址，除非用户明确要求且目标地区/岗位有正当需要。
- 头像默认省略；不把中国中文简历的头像习惯迁移到英文简历，也不从模板头像或示例图片自动填充。

隐私字段属于 `profile` claim，同样需要来源、置信度和用户确认。生成 HTML 时遵循用户选择；本文件不要求或授权修改 HTML 模板。

## 七、板块排序与投递建议

基础简历应先保证事实完整，再根据岗位相关性调整顺序。应届生通常可将教育背景和项目经验放在前面；有强相关工作经历时按岗位需要前置。不要为了追逐关键词删除能证明能力的事实。

文件名保持清晰、可检索，例如：

```text
姓名_岗位_版本.pdf
```

投递前确认目标平台的文件格式、大小、命名和隐私要求。针对同一 JD 的定制版应保留基础简历，并记录改动点、关键词来源、未匹配要求和待确认字段。

## 八、投递前检查清单

### 事实与证据

- [ ] 每个 `profile`、`education`、`experience`、`projects`、`campus`、`skills`、`awards`、`summary` claim 都能追溯到 `source`。
- [ ] 每个 claim 都有简短 `evidence`；团队成果与个人贡献已区分。
- [ ] `confidence` 为 `medium/low` 的字段已向候选人回访，或未进入最终输出。
- [ ] 所有数字、比例、日期、范围和结果都标注 `metric_status`；未验证指标没有悄悄写入。
- [ ] 没有新增雇主、职位、技能、日期、数字、因果关系或能力等级。

### 写作与隐私

- [ ] 每条经历以清晰行动和真实角色为主，结果优先使用已核验指标；无数字时使用规模/范围/时间/质量/角色/交付物证据。
- [ ] 没有把 STAR、数字数量、自我评价字数或技能数量当成机械硬指标。
- [ ] `summary` 只归纳其他板块已有事实，没有空洞自评或新主张。
- [ ] 已按中文或英文投递场景检查隐私字段；头像遵循“用户明确提供并同意才展示”的默认策略。

### 关键词与输出模式

- [ ] 每个关键术语标注来源（`candidate`、`jd` 或 `user_preference`）。
- [ ] JD 关键词只用于对齐和排序，没有被写成候选人事实；缺口已如实记录。
- [ ] 已明确选择视觉 HTML/PDF 或 ATS-safe 模式，且没有混用两种模式的呈现假设。
- [ ] 视觉模式已检查打印页数、裁切、复制文本和链接；ATS-safe 模式已检查单栏阅读顺序、标准标题和纯文本可解析性。
- [ ] 已根据目标平台确认格式、大小、命名和提交要求；没有承诺 ATS 必然通过。
- [ ] 待确认字段只出现在采集/分析报告，最终简历 HTML/PDF 中没有待确认内容。
