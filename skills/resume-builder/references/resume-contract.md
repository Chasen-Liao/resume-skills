# 简历事实契约

本契约是 `resume-builder` 与 `jd-tailorer` 共用的事实边界。它规定简历可以记录什么、如何记录证据，以及如何在不改变事实的前提下改写内容。它不替代 HTML 模板，也不决定视觉样式。

## 一、不可逾越的边界

- 只使用候选人明确提供、现有简历明确写出，或候选人可核验材料明确支持的事实。
- 改写只能改变表达、顺序、粒度和与 JD 的术语对齐，不得新增或推断雇主、职位、日期、学历、技能、数字、成果、职责、奖项或用户/业务规模。
- JD 中的要求、关键词和公司描述只是目标岗位信息，不是候选人事实。不能因为 JD 提到某项技术，就把它写进候选人的技能或经历。
- 不能用“听起来合理”的数字填补空白。没有可信来源的数字必须追问、标记为未验证，或从最终输出中删除。
- `confidence` 反映事实被当前证据支持的程度，不等于允许代理猜测；低置信度内容不能以确定口吻输出。
- 不确定的个人隐私字段默认不收集、不展示；若用户明确要求展示，也应先确认准确性、用途和投递范围。

## 二、简历数据结构

简历母版可以包含以下板块；没有事实支撑的板块保持为空，不用占位内容凑数。`resume-builder` 和 `jd-tailorer` 必须使用这里的 `section` 枚举检查每条 claim：

| 板块 | 允许记录的内容 |
|---|---|
| `profile` | 姓名、联系方式、所在地、求职意向、个人链接，以及用户确认需要展示的工作授权信息 |
| `education` | 学校、专业、学历、时间、成绩、课程、学术活动；仅写用户能确认的内容 |
| `experience` | 实习、工作、研究或其他正式经历中的组织、角色、时间、行动、产出和结果 |
| `projects` | 课程、个人、开源、研究或竞赛项目中的角色、范围、技术、交付物和结果 |
| `campus` | 校园组织、学生工作、志愿服务、社团、校园活动或其他校园经历中的组织/活动、角色、时间、行动、交付物和结果 |
| `skills` | 用户实际使用或明确掌握的技术、工具、语言、方法和能力；熟练度必须有用户依据 |
| `awards` | 奖项、竞赛、证书、荣誉、级别、授予方和时间；不得把参与写成获奖 |
| `summary` | 由上述已验证事实归纳出的职业定位、相关能力和求职方向，不引入新事实 |

每条对外可见的陈述都视为一个 `claim`。两个入口都应在采集/分析阶段按以下记录保存或检查；没有这些字段就不能进入最终简历：

```yaml
section: experience        # profile/education/experience/projects/campus/skills/awards/summary
claim: "优化数据查询流程并完成上线交付"
source: user               # user/resume/artifact/derived
confidence: high           # high/medium/low
evidence:
  - "候选人说明其负责查询流程优化"
  - "候选人提供上线记录或可核验项目材料"
metric_status: verified    # verified/estimated/unverified/not_applicable
```

### 字段定义

- `source`：事实来源。`user` 是候选人直接确认；`resume` 是已有简历原文；`artifact` 是作品、代码、成绩单、证书、上线记录等材料；`derived` 只能表示对已有事实的保守归纳，不能表示新事实来源。JD、市场文章和模板示例不得作为候选人事实来源。
- `confidence`：`high` 表示来源清楚且前后一致；`medium` 表示有来源但仍需候选人确认；`low` 表示来源模糊、可能过时或存在冲突。`medium/low` 内容在输出前应回访确认。
- `evidence`：能解释“为什么可以这样写”的原始事实、材料或可核验线索。不要把改写后的句子本身当作证据。
- `metric_status`：`verified` 表示数字、比例、时间、范围等指标有明确来源；`estimated` 只适用于候选人明确说明是估算并同意这样呈现的情况；`unverified` 表示待确认，不能直接进入成稿；`not_applicable` 表示该 claim 没有适用的数字指标。

### 契约检查清单

- [ ] 每条 claim 的 `section` 属于 `profile`、`education`、`experience`、`projects`、`campus`、`skills`、`awards` 或 `summary`。
- [ ] `campus` claim 已记录可核验的组织/活动、角色、时间、行动、交付物或结果，并具有同样的 `source`、`confidence`、`evidence`、`metric_status` 字段。
- [ ] 所有 section 都只保留有事实支撑的内容；待确认字段仍停留在采集/分析记录。

## 三、改写规则

1. 先拆出事实，再写句子：区分背景、候选人行动、产出、结果和证据，不把职责自动升级为成果。
2. 优先追问可核验指标，例如处理规模、时间范围、性能、质量、完成状态或影响范围；指标没有来源时不补写。
3. 没有可信数字时，允许使用已确认的非数字证据：规模层级或范围、时间节点、质量/验收/上线状态、交付物，以及候选人的真实角色（如独立、主导、协作）。这些证据不能被夸大成数字或因果成果。
4. `jd-tailorer` 只能在已有 claim 中重排和对齐 JD 关键词；关键词必须标注来源，且不得制造缺失经验。无法匹配的要求应列为缺口。
5. `summary` 只能是已有板块的归纳；不能凭空增加“精通”“专家”“高增长”等强度判断。
6. 在视觉 HTML/PDF 与 ATS-safe 输出之间切换时，只改变呈现方式，不改变事实、证据状态或未验证标记。

## 四、输出前的最小门槛

一个 claim 只有在以下条件满足时才能进入最终简历：

- 有 `source` 和可理解的 `evidence`；
- `confidence` 不为 `low`，或已获得候选人确认；
- 若包含指标，`metric_status` 为 `verified`，或明确标注为候选人认可的估算；
- 改写没有引入来源中不存在的新实体、数字、因果关系或能力等级；
- 若用于 JD 定制，使用的关键词能追溯到 JD 或候选人已有材料。

不满足门槛的内容应进入“待确认字段”清单，而不是悄悄补全或以确定语气发布。待确认字段只能出现在采集记录、匹配分析或其他明确标为待确认的分析报告中；不得进入最终简历成稿，包括视觉 HTML/PDF 和 ATS-safe HTML/PDF。
